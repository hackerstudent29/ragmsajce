Build a complete, production-grade ingestion, structuring, normalization, and retrieval-ready data pipeline for the MSAJCE AI system.

This system MUST process BOTH:

1. Website links (all provided URLs)
2. Manually provided structured data (JSON like transport routes, personal profile, etc.)

The system must NOT directly store or embed raw data.

It must follow a strict multi-stage pipeline:
SCRAPE → CLEAN → FILE STORE → GLOBAL ANALYSIS → SPLIT → ALIGN → NORMALIZE → STRUCTURE → VALIDATE → CHUNK → EMBED → STORE

---

## CORE OBJECTIVE

* Handle ALL domains (person, transport, departments, contact, general, etc.)
* Avoid focusing on any single data type
* Convert all data into query-efficient structure
* Ensure zero hallucination and high accuracy
* Work like ChatGPT/Gemini but grounded in your DB

---

## STEP 1 — SCRAPE ALL LINKS

From all given URLs extract:

* title
* headings (h1, h2, h3)
* paragraphs
* tables
* lists

Remove:

* navbar
* footer
* scripts
* ads
* duplicate content

---

## STEP 2 — STORE RAW DATA INTO FILES

DO NOT process immediately.

Store each page into:
/raw_data/{page_name}.json

Format:
{
url,
title,
headings,
content,
tables,
extracted_at
}

---

## STEP 3 — CLEAN DATA

Create cleaned version:
/clean_data/{page_name}.json

Apply:

* remove noise
* remove repeated lines
* normalize spacing
* remove unnecessary symbols

---

## STEP 4 — LOAD MANUAL DATA

Load manually provided data (transport routes, personal profile, etc.)

Store temporarily:
/manual_data/master.json

---

## STEP 5 — GLOBAL ANALYSIS (CRITICAL)

Load ALL data together (clean_data + manual_data)

Analyze relationships:

* match people across pages
* connect departments with HODs
* connect transport routes with stops and drivers
* merge duplicate or overlapping information

Goal:
Create a GLOBAL understanding across ALL sources

---

## STEP 6 — SPLIT INTO ATOMIC RECORDS

Break nested data into smallest usable units:

* each person → one record
* each route → one record
* each stop → one record
* each contact → one record
* each department → one record

DO NOT keep large nested objects.

---

## STEP 7 — DATA ALIGNMENT

Reorganize data by meaning (NOT by page)

Group into unified categories:

* people
* departments
* transport routes
* transport stops
* public transport (MTC)
* contacts
* facilities
* general info

Link related data:

* route ↔ stops
* route ↔ driver
* department ↔ HOD
* person ↔ role

---

## STEP 8 — NORMALIZATION

Normalize all key fields:

* lowercase
* remove titles (dr, mr, prof)
* remove punctuation
* trim spaces

Apply to:

* names
* roles
* departments
* locations

---

## STEP 9 — ALIAS GENERATION

For every person and entity:

Generate:

* short names
* last names
* alternate forms

Example:
"Dr. K.S Srinivasan" →
normalized_name: "ks srinivasan"
aliases: ["srinivasan", "ks srinivasan"]

---

## STEP 10 — FINAL STRUCTURED FILES

Create:

/structured_data/

people.json
departments.json
transport_routes.json
transport_stops.json
mtc_routes.json
contacts.json
general.json

---

## STEP 11 — VALIDATION

Before storing:

* remove duplicates
* remove empty data
* ensure every record is complete
* ensure relationships are valid

---

## STEP 12 — DATABASE STORAGE

Store into MongoDB:

1. entities_master → people
2. transport_routes → route info
3. transport_stops → stops
4. mtc_routes → public transport
5. structured_data → departments, contacts, facilities
6. vector_store → ONLY general descriptive text
